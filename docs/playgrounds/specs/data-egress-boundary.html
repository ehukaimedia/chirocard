ChiroCard · Architecture Playground · Target Contract

The Data-Egress Boundary

ChiroCard's wedge is "your health record is yours and stays on your device." This playground pins the single contract that makes that claim true and enforceable: a hard boundary around the device, an explicit inventory of every point where data crosses it, a named owner for each, and a consent model. Everything else in the privacy remediation flows from this boundary.

active specs/ · target-contract informs Audit Finding 1 (Critical)
Decision this informs. How to resolve the contradiction in Audit Finding 1 — privacy claims that the code does not enforce.
Anti-pattern it prevents. Shipping "zero-knowledge / never leaves your device / HIPAA-GDPR compliant" copy while Google Tag Manager (hardcoded container), Photon address search, and a public CORS proxy send data off-device with no consent gate. A claim that is true regardless of the facts is a bug, not a feature.

Edge-direction evidence: the GTM <script> in index.html:14 is the emitter to googletagmanager.com; analytics.ts:5-9 pushes to window.dataLayer (the repo-verifiable boundary). The container's configured target is GA4 — the GA4 event tags are provisioned by scripts/setup_chirocard_tags.js — but what the live container forwards is config-dependent and external to this repo. places.ts:65 and googleMaps.ts:46 are the literal fetch() call sites.

Runner-up contract considered & rejected for this playground: the QR/pako session wire format. It would be a real contract deserving a verification gate — but it is currently not implemented (Audit Finding 15: compressData/decompressData have no callers; the QR libs are unused). So it is neither an egress concern nor a present test target; a golden round-trip + corruption gate belongs with that feature if/when it is built. The QR edge appears above only to mark the intended peer flow as designed-but-absent.
Rejected remediation option: "keep the public allorigins.win proxy and just add a sentence to the Privacy page." Rejected — a health app should not route any user-supplied URL through an arbitrary public proxy; disclosure does not remove the third-party-in-the-path risk.

Public interfaces (single source of truth)

  • src/lib/consent.ts — owns consent state ("unset" | "granted" | "denied"), persisted in IndexedDB/localStorage; the only module that authorizes analytics.
  • loadAnalytics() — idempotent deferred GTM loader, called only when consent is granted; reads container id from import.meta.env.VITE_GTM_ID.
  • trackEvent() (src/utils/analytics.ts) — gains a consent check; becomes a no-op otherwise.
  • EGRESS_ALLOWLIST — one constant that is the source of truth for both the CSP header and the Privacy-page third-party list (docs and enforcement cannot drift).

Ownership boundaries

  • Device / IndexedDB — patient owns it. db.ts is the single schema source of truth (Dexie v17 + migrations); records are written through that Dexie instance from many components (e.g. PractitionerManager.tsx:66, AddJournalModal.tsx:17, RoutineVerificationModal.tsx:71) and the WebDB.ts adapter — not a single writer. The contract invariant is only that these records are never emitted off-device.
  • Analytics egress — consent module authorizes; deferred loader emits; Google processes.
  • Address egress — places.ts + AddressAutocomplete.tsx own; fires on debounced practitioner input. Currently also emits device GPS lat/lon — to be dropped or consent-gated.
  • Link-resolution egress — googleMaps.ts owns; flagged for the remove/replace decision.

Data flow (target)

  • App boot → render → no analytics. Consent prompt shown once.
  • Consent granted → loadAnalytics() injects GTM (env id) → later trackEvent() calls flow to GA4.
  • Consent denied/unset → GTM never loads; trackEvent() is a no-op; records never leave regardless.
  • Practitioner address input → debounced places.ts call to Photon (disclosed).

Build sequence (what the plan implements, in order)

  • 1 — Land this boundary doc (done).
  • 2 — consent.ts + consent UI; move GTM out of index.html into a gated loadAnalytics().
  • 3 — Container id from VITE_GTM_ID; add CSP from EGRESS_ALLOWLIST.
  • 4 — Resolve the allorigins.win decision (prefer remove).
  • 5 — Rewrite Privacy page + README copy to match the boundary exactly; drop "Vercel Analytics".

Acceptance tests (the gate — must pass)

[GATE-1] No-consent ⇒ no telemetry. With consent unset/denied, zero requests to googletagmanager.com / google-analytics.com fire (assert in a test / observe in network panel).
[GATE-2] No hardcoded container. grep "GTM-" over index.html/src returns nothing; the id resolves from env at build.
[GATE-3] Docs match code. The Privacy page's third-party list equals EGRESS_ALLOWLIST exactly — no "Vercel Analytics", no omissions.
[GATE-4] CSP constrains egress. A CSP restricts script-src/connect-src to the allowlisted domains; a request to a non-allowlisted origin is blocked.
[GATE-5] Records stay local. No IndexedDB record content appears in any outbound request body across the core flows.

Non-goals

  • The QR/pako wire-format schema and its corruption tests (separate test target — plan §C).
  • UI/visual redesign, bundle-splitting, or any non-egress finding.
  • Building a first-party analytics backend — out of scope; consent-gated GA4 is acceptable if disclosed.
